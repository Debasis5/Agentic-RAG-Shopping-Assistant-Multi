# Migration Plan: Single-graph pipeline → Supervisor-orchestrator multi-agent

**Goal:** Replace the monolithic LangGraph pipeline with a supervisor graph that orchestrates
three independent sub-agent graphs (RAG, Order, Escalation).

**Test strategy:** Each step is tested in `notebooks/test_notebook_multi.ipynb` before moving to the next.
The original `notebooks/test_notebook.ipynb` is preserved and untouched.

## Step details

### Step 1 — States

**Files to create:**
- `src/supervisor/state.py` — `SupervisorState`
- Sub-agent states defined inline in each agent file (Steps 2–4)

**`SupervisorState` fields:**
```python
query: str
messages: Annotated[list, add_messages]
guardrail_decision: str   # "PASS" | "BLOCK"
agent_outcome: str        # which agent was called: "rag" | "order" | "escalation" | "chitchat"
agent_response: str       # raw response returned by the sub-agent
final_response: str       # after synthesis
```

**Note:** `src/state.py` will re-export `SupervisorState` as `GraphState` for backwards compatibility
with `api.py` and `main.py` (updated in Step 8).

### Step 4 — Escalation agent

**File:** `src/agents/escalation_agent.py`

**Graph:**
```
complaint_handler → human_handoff → ticket_creation → END
```

**State (`EscalationAgentState`):**
```python
query: str
complaint_type: str     # classified by complaint_handler
ticket_id: str          # generated by ticket_creation
agent_response: str
```

**New nodes to create** (all mock, no real APIs):
- `complaint_handler` — LLM classifies complaint type (damaged item / wrong item / late delivery / other)
- `human_handoff` — generates a handoff message with support number + estimated wait time
- `ticket_creation` — generates a mock ticket ID and confirmation message

**Notebook section:** Step 11 — Escalation Agent

---

### Step 5 — Delegation router

**File:** `src/supervisor/delegation_router.py`

**Replaces:** `src/nodes/intent_router.py`

**LLM classifier** — maps query to one of: `rag | order | escalation | chitchat`

| Intent | Trigger |
|--------|---------|
| `rag` | policy/FAQ questions (returns, payments, shipping, product conditions, account) |
| `order` | specific order/shipment/return/account status — needs live data |
| `escalation` | complaints, damaged items, wrong item delivered, requests for human agent |
| `chitchat` | greetings, small talk, thanks |

**Notebook section:** Step 12 — Delegation Router

---

### Step 6 — Supervisor synthesis

**File:** `src/supervisor/synthesis.py`

**Responsibilities:**
- Receives `agent_response` from whichever sub-agent ran
- For `rag` path: runs a faithfulness check (does the answer stay within the retrieved docs?)
- For `order` / `escalation` / `chitchat` paths: passes through directly
- Sets `final_response` on `SupervisorState`

**Notebook section:** Step 12 — Supervisor Synthesis

---

### Step 7 — Supervisor graph

**File:** `src/supervisor/graph.py`

**Graph:**
```
guardrail → BLOCK → END
          → PASS  → delegation_router
                        → rag_agent.invoke()       → synthesis → END
                        → order_agent.invoke()     → synthesis → END
                        → escalation_agent.invoke()→ synthesis → END
                        → chitchat_node            → synthesis → END
```

**Note:** Sub-agents are invoked via `.invoke()` inside a wrapper node in the supervisor graph,
not as nested LangGraph subgraphs. This keeps streaming simple.

**Notebook section:** Step 13 — Full supervisor end-to-end

## Decisions log

| Date | Decision | Reason |
|------|----------|--------|
| 2026-06-04 | RAG agent keeps bare vector lookup (no reranker/query rewriter) | Complexity deferred; sub-agent boundary makes future upgrade a contained change |
| 2026-06-04 | Chitchat handled inline in supervisor (no sub-agent graph) | Doesn't need its own state or tools; adding a graph would be overhead with no benefit |
| 2026-06-04 | Guardrail stays in supervisor only | Runs once before delegation; sub-agents should never receive blocked queries |
| 2026-06-04 | Sub-agents invoked via `.invoke()` wrapper nodes, not nested subgraphs | Keeps SSE streaming in api.py straightforward |
| 2026-06-04 | Existing `src/nodes/` files reused as-is | Avoids unnecessary rewrites; nodes are already correct and tested |
